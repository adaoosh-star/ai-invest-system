## 🔍 问题根因

### 核心问题

**工具输出缓冲区溢出**

**机制：**
1. OpenClaw 工具输出有硬限制（约 50KB 或 2000 行）
2. 运行了大量测试命令，输出累积超过限制
3. 后续所有工具输出被截断
4. 对话通道不受影响（双通道隔离）

**证据：**
- `echo "TEST"` 也被截断
- 子会话能正常输出（新会话，空缓冲区）
- 对话通道一直正常
- 昨天也出现过类似问题，后来自动恢复

## ❌ 错误做法

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 反复执行大量测试命令 | 输出累积，撑满缓冲区 | 用 `head -10` 限制输出 |
| 输出完整内容到工具通道 | 占用大量缓冲区 | 大输出保存到文件 |
| 用主会话执行测试 | 主会话缓冲区满 | 用子会话执行测试 |
| 没有及时限制输出大小 | 超过系统限制 | 一开始就加限制 |
| 推测原因而不是查证 | 浪费时间 | 先查文档或问官方 |

## 💡 关键教训

### 1. 输出大小要控制

**教训：**
- 工具输出缓冲区有限制
- 大量输出会撑满缓冲区
- 一旦满了，所有工具输出被截断

**改进：**
```bash
# 之前（错误）
python3 utils/report_generator.py 000788.SZ
# 输出大量内容 → 缓冲区满 → 截断

# 之后（正确）
python3 utils/report_generator.py 000788.SZ > /tmp/report.txt && echo "完成"
# 输出到文件 → 只打印"完成" → 不占缓冲区
```

---

### 2. 用子会话执行测试

**教训：**
- 主会话缓冲区满了会影响所有工具输出
- 子会话有独立缓冲区
- 子会话能正常输出时，说明是主会话问题

**改进：**
```bash
# 之前（错误）
# 在主会话反复执行测试命令

# 之后（正确）
# 用 sessions_spawn 创建子会话执行测试
```

---

### 3. 先查文档再推测

**教训：**
- 我推测了"工具输出缓冲区"、"50KB 限制"、"TTL 自动清理"
- 但这些都是基于一般系统设计的推测
- 不是基于 OpenClaw 的实际文档或配置

**改进：**
- 遇到问题先查官方文档
- 或者问系统管理员
- 不要自己推测，可能完全错误

---

### 4. 及时保存工作成果

**教训：**
- 工具输出被截断时，无法验证代码是否正常
- 但代码已经写在磁盘上，不会丢
- 工作区文件一直安全

**改进：**
- 代码及时提交到 git
- 报告及时保存到文件
- 不依赖工具输出验证

---

### 5. 识别系统级问题

**教训：**
- 当 `echo "TEST"` 也被截断时，应该意识到是系统级问题
- 不是代码问题，不是逻辑问题
- 是 OpenClaw 工具输出通道问题

**改进：**
- 简单命令也被截断 → 系统级问题
- 停止调试，等待系统恢复
- 或者切换到其他方式（如子会话）
